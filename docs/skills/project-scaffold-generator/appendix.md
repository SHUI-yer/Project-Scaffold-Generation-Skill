# 通用项目脚手架生成器（Project Scaffold Generator）

## 目标

根据工作目录中的需求文档（.md/.txt/.docx）或结构化问答，输出可直接运行的完整项目代码与交付文档，并在生成过程中分层确认与质量闸口检查。

## 何时调用

- 用户提出“生成项目脚手架/模板/初始化工程/从需求生成代码”等需求
- 用户提供需求文档，希望自动提取模块并生成工程
- 用户没有文档，希望通过问答快速梳理需求并生成工程

## 小白友好沟通规范

- 若用户表现出“第一次做项目/不懂术语/希望讲简单一点”，自动切换到小白解释模式
- 术语第一次出现时，必须用一句通俗话解释“它是什么、解决什么问题”，再继续推进
- 给技术选项时，必须同时说明“适合场景、优点、代价/缺点、不推荐场景”
- 给推荐方案时，必须说明“为什么推荐它、为什么不是另一个、它的学习成本和维护成本”
- 尽量少用纯缩写；如必须使用，优先写成“JWT（登录令牌）”“ORM（数据库对象映射工具）”这类形式
- 若用户是小白，默认先给结论，再给解释，再给推荐，不要直接扔一组黑话
- 涉及多方案比较时，优先输出简明对比表或短列表，让用户能快速理解差异

## 交互原则（强约束）

- **所有提问必须使用 AskUserQuestion 工具以选项形式呈现，严禁要求用户手动打字输入**
- 每轮提问最多 4 个问题，每个问题提供 2-4 个选项（含推荐标记）
- 多语言适配：如用户未指定输出语言，在 Phase 1 先确认输出语言（中文/英文/双语）
- 默认输出可运行代码（非伪代码），并生成 README.md、.gitignore
- 支持“分阶段构建顺序”选择：先让用户选择科学的构建路径，再按顺序逐段生成、逐段确认
- 若用户对某个技术项不理解，先解释术语与差异，再继续问答，不强推其立即决策
- 代码无多余注释、无硬编码密钥/密码；敏感信息使用 .env / 配置文件占位 + 示例文件
- 后端严格分层：Controller → Service → Mapper/Repository
- 前端强类型：TypeScript interface 定义所有数据结构
- 安全基线随"项目安全性等级"动态提升：所有等级都必须具备参数校验、密钥外置与 SQL 注入防护；更高等级自动叠加权限、限流、审计、环境隔离等能力
- 统一响应：Result{ code, message, data }
- 统一异常处理：GlobalExceptionHandler（或等价机制）
- **认证失败响应规范（强制）**：
  - 未认证请求 → HTTP 401 + JSON `{"code":401,"message":"未登录或登录已过期"}`
  - 已认证但无权限 → HTTP 403 + JSON `{"code":403,"message":"权限不足"}`
  - 不得返回 HTML 错误页面，所有错误必须返回 JSON
  - Java Spring Security：必须配置 `AuthenticationEntryPoint` 返回 401
  - Python/FastAPI：必须配置 `HTTPException` handler 返回 401
  - Node.js/Express：必须配置 JWT error handler 返回 401

## 支持范围（可扩展）

- 后端语言：Java / Python / JavaScript(TypeScript) / Go / Rust
- 架构：前后端分离 / 单体应用 / 纯 API 服务 / CLI 工具
- 数据库：MySQL / PostgreSQL / MongoDB / SQLite / H2
- 前端：Vue 3 / React / Svelte / 无前端

## Phase 流程（7 个 Phase）

### Phase 1：需求获取

1) 扫描工作目录：
- 优先匹配：README*、需求*、PRD*、spec*、requirements*、需求文档*
- 后缀优先级：.md > .txt > .docx

2) 解析策略：
- .md/.txt：直接读取并做结构化抽取
- .docx：优先尝试提取纯文本（若运行环境不具备解析能力，提示用户将 docx 转为 md/txt 或允许安装解析依赖后再继续）

3) 抽取字段（尽量从文档推断，推断必须标注“待确认”）：
- 项目类型（Web/CLI/API/单体/前后端分离）
- 功能模块（按模块名+功能点列出）
- 用户角色与权限（角色清单、可访问模块）
- 数据模型（实体、字段、关系、约束）
- 技术偏好（语言、框架、数据库、部署偏好）
- 项目安全性等级（个人开发级 / 开源发布级 / 标准业务级 / 企业级）
- 业务规则（校验规则、状态流转、边界条件）
- 非功能需求（性能、并发、日志、审计、国际化）

4) 若未找到任何文档：
- 进入 Phase 3 的结构化问答（仍然先执行 Phase 2 的需求表格，但大部分标为“待确认”）

### Phase 2：需求分析（结构化表格输出）

输出一张结构化需求表（Markdown 表格），字段固定为：
- 项目类型
- 功能模块
- 用户角色
- 数据模型
- 技术栈（后端/前端/数据库/ORM/鉴权）
- 安全等级
- 部署（本地/生产/Docker/CI）
- 安全（认证/鉴权/校验/限流）

每个单元格包含：
- 已确定：直接填值
- 待确认：填“待确认（推断：xxx）”或“待确认（缺失）”

### Phase 3：细节追问（5 轮，每轮最多 4 题）

- 每题 2–4 个选项（可含“其他（我来补充）”）
- 根据已抽取信息动态删减问题：已确定则不问
- 推荐项放在选项第一位并标注“（推荐）”

## 首次运行强制标准（生成即跑，不许手动补救）

**核心原则：生成的项目必须在用户执行一次安装命令（如 `pip install -r requirements.txt` + `python main.py`）后直接运行，不得要求用户手动修改代码或手动执行额外 SQL。**

### 通用强制项（所有技术栈）

| # | 检查项 | 为什么重要 |
|---|--------|-----------|
| 1 | 入口文件有完整启动代码 | 没有启动代码 = 空壳项目 |
| 2 | 所有 import 路径可解析 | import 不存在的模块 = ImportError 直接崩溃 |
| 3 | 依赖清单完整 | 缺包 = ModuleNotFoundError |
| 4 | Python 包有 `__init__.py` | 缺 init = 包无法被识别 |
| 5 | 数据库首次运行自动建表 | 不自动建表 = 用户必须手动执行 SQL |
| 6 | .env 加载代码存在 | 有 .env.example 但没加载代码 = 配置不生效 |
| 7 | CORS 已配置（前后端分离） | 没 CORS = 前端请求全部被浏览器拦截 |
| 8 | 跨层调用链完整 | 方法不存在 = 运行时报 AttributeError |
| 9 | 配置文件可读取 | 格式不匹配 = 配置加载失败 |
| 10 | 默认端口不冲突 | 占用系统端口 = 启动失败 |

### 技术栈特定强制项

**Java Spring Boot：**
- pom.xml 中 spring-boot-starter-web 和数据库驱动必须存在
- application.yml 中数据库配置与 .env 对应
- 认证入口点配置（401 + JSON）

**Python 桌面应用（Tkinter）：**
- main.py 中 mainloop() 必须存在
- 数据库连接使用单例模式

**Python Web（FastAPI/Django）：**
- uvicorn/django 在依赖中
- CORS 中间件已配置

**Node.js Express：**
- package.json 中 scripts.start/dev 存在
- cors 包在依赖中

**React/Vue 前端：**
- scripts.dev/build 存在
- API 请求使用环境变量 base URL
