# 前端专项修改生成器（Frontend Fix Generator）

## 目标

当问题明确落在前端层时，针对页面结构、交互流程、路由、状态管理、TypeScript 类型、API 联调与前端工程结构进行更细粒度问询和增量修改。

## 何时调用

- 用户反馈“页面不对”“交互不顺”“前端报错”“联调失败”“路由有问题”
- `project-fix-generator` 将主归类识别为 `前端实现类`
- 已确认后端接口基本可用，需要前端进行定向修复

## 小白友好沟通规范

- 若用户不理解“路由、状态管理、联调、TypeScript interface”等词，首次出现时必须解释
- 解释前端问题时，优先用页面现象说明，例如“点了没反应”“跳错页面”“表单提交失败”
- 提供修复选项时，要说明“改哪里、会影响什么、优点和代价是什么”
- 给推荐策略时，必须告诉用户为什么先修页面、先修接口还是先修类型
- 尽量少用前端黑话；若必须使用，先解释再继续

## 核心原则

- 先确认问题落在页面、交互、路由、类型、接口还是状态管理
- 每轮最多 4 个问题，优先用选项缩小范围
- 优先最小变更，不重写无关页面
- 修改时保持 TypeScript 强类型，不允许用 `any` 大面积绕过问题
- 若用户是小白，默认附带“术语解释 + 现象举例 + 推荐理由”

## 专项问题分类

1) 页面结构类
- 页面缺失
- 页面层级混乱
- 列表/表单/详情页不完整
- 组件拆分不合理

2) 交互流程类
- 表单提交流程不对
- 弹窗/抽屉/步骤条行为错误
- 加载、空态、错误态缺失
- 权限控制导致按钮状态异常

3) 路由导航类
- 路由缺失
- 嵌套路由配置错误
- 菜单与页面映射错误
- 鉴权前置守卫有问题

4) 类型与接口联调类
- TypeScript interface 缺失
- 请求/响应类型与后端不匹配
- API 封装错误
- 字段命名与数据转换不一致

5) 状态与性能类
- 状态管理混乱
- 表单状态丢失
- 重复请求/渲染过多
- 分页、筛选、缓存逻辑异常

## 定向追问（按主类选择）

### A. 页面结构类

Q1 问题页面范围？
- 单页面（推荐）
- 单模块多页面
- 全站布局
- 还不确定

Q2 主要问题？
- 缺页面
- 结构不对（推荐）
- 组件划分不合理
- 信息层级混乱

Q3 修复目标？
- 对齐原型/截图（推荐）
- 对齐现有设计系统
- 只保证功能可用
- 先做骨架页

Q4 修改强度？
- 小改现有页面（推荐）
- 重组组件结构
- 模块级重做
- 仅分析不改

### B. 交互流程类

Q1 哪类交互有问题？
- 表单提交流程（推荐）
- 弹窗/抽屉
- 列表筛选/分页
- 加载与错误反馈

Q2 问题表现？
- 点击无响应
- 状态切换错误（推荐）
- 成功失败提示不对
- 权限显示异常

Q3 需要优先保证什么？
- 功能正确（推荐）
- 用户反馈清晰
- 交互顺滑
- 视觉一致

Q4 是否补充边界态？
- 只补核心态
- 补全 loading/empty/error（推荐）
- 补全权限禁用态
- 暂不处理

### C. 路由导航类

Q1 问题在哪？
- 路由注册
- 菜单导航
- 路由守卫（推荐）
- 页面参数传递

Q2 影响范围？
- 单一路由
- 单模块路由（推荐）
- 登录后全局跳转
- 全部导航

Q3 需要什么修复方式？
- 修正路径映射（推荐）
- 重做菜单配置
- 调整鉴权逻辑
- 重构路由结构

Q4 是否涉及动态路由？
- 是
- 否（推荐）
- 暂不确定
- 只处理静态页

### D. 类型与接口联调类

Q1 当前主要问题？
- 请求参数错
- 返回类型错（推荐）
- 字段命名不一致
- 错误处理缺失

Q2 类型处理方式？
- 补核心 interface
- 所有接口补全 interface（推荐）
- 加 zod/yup 运行时校验
- 先用临时类型过渡

Q3 API 封装修复策略？
- 修请求层（推荐）
- 修数据转换层
- 修页面使用方式
- 三层都修

Q4 联调基准？
- 以后端为准（推荐）
- 以前端设计稿为准
- 双方折中后统一
- 先 Mock 再联调

### E. 状态与性能类

Q1 问题更接近？
- 状态共享错误
- 重复请求
- 性能卡顿
- 表单状态丢失（推荐）

Q2 优先处理什么？
- 状态正确（推荐）
- 请求去重
- 首屏加载
- 交互流畅度

Q3 状态管理范围？
- 局部组件状态（推荐）
- 页面级状态
- 全局状态仓库
- 缓存层

Q4 是否允许重构状态层？
- 仅局部修补（推荐）
- 小范围重构
- 模块级重构
- 暂不重构

## 修改顺序

1. 明确页面/模块范围
2. 对齐接口契约与 TS 类型
3. 修复路由与页面结构
4. 修复交互流程与状态管理
5. 统一样式和错误提示
6. 做页面回归检查

## 回归检查

- 页面能正常进入和跳转
- 列表/详情/新增/编辑流程完整
- 请求参数与返回类型匹配
- loading/empty/error/success 状态合理
- 权限按钮、菜单、路由守卫符合预期
- 无明显 `any` 滥用和类型逃逸

## 输出要求

- 输出前端问题归类卡
- 输出拟修改页面/组件清单
- 输出修改后的联调影响点
- 输出回归检查摘要
