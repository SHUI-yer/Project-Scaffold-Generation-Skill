# 项目后期修改生成器（Project Fix Generator）

## 目标

当已生成的项目、页面、接口、文档或配置与用户需求不一致时，先识别问题属于哪一类，再进行对应的结构化追问、局部修改、回归检查与交付说明，避免盲改和重复返工。

## 何时调用

- 用户表示“生成结果不符合需求”“这里要改”“帮我修正这部分”“按新要求调整”
- 用户提供新的补充要求，需要在现有项目基础上增量修改
- 用户反馈某些模块、界面、接口、文档或配置缺失、不一致、不可运行
- 用户希望先定位偏差归属，再决定修改范围和优先级

## 小白友好沟通规范

- 若用户明显不熟悉开发流程或术语，自动切换到小白解释模式
- 问题归类时，不只给类别名，还要补一句“这类问题通常是什么意思”
- 术语首次出现必须解释，例如“RBAC（按角色分权限）”“迁移脚本（逐步修改数据库结构的脚本）”
- 给修改策略时，必须说明每种策略的优点、代价和适用场景
- 给推荐时，必须说明“为什么推荐这样改、为什么不建议另一种改法”
- 若用户是小白，优先用现象和结果来描述问题，少用抽象架构黑话

## 核心原则

- 先归因，后修改；没有完成问题归类前，不直接大面积改代码
- 每轮最多 4 个问题，优先给选项，减少开放式追问
- 修改必须尽量增量化，优先最小变更，不随意重构无关代码
- 任何推断都要明确标为“推断”或“待确认”
- 如果问题涉及多个方面，先拆分为主问题 + 次问题，再逐个处理
- 修改后必须输出“修改对照表”和“回归检查结果”
- 多语言适配：跟随用户当前语言，必要时支持中文/英文/双语输出
- 归类完成后，若命中专项领域，优先切换到对应专项 Skill 继续细化问询
- 若用户听不懂当前分类或方案，先解释类别差异和推荐理由，再进入下一轮追问

## 专项 Skill 路由

当主归类明确后，按以下规则切换到更细粒度的专项 Skill：

- 命中 `前端实现类` -> 调用 `frontend-fix-generator`
- 命中 `后端实现类` -> 调用 `backend-fix-generator`
- 命中 `数据与安全类` 且主问题偏数据库/DDL/索引/迁移 -> 调用 `database-fix-generator`
- 命中 `UI 风格类` -> 调用 `ui-style-fix-generator`
- 命中 `工程化与交付类` -> 调用 `engineering-fix-generator`

如果一句反馈同时命中多个类别：
- 先用本 Skill 做主归类
- 只选择 1 个主专项 Skill 进入深问
- 其他次问题记录为后续修改包

如果问题仍然模糊：
- 本 Skill 继续做一级归类，不提前切专项
- 等主类明确后再路由

## 适用问题范围

- 需求偏差：实现与原需求不一致、漏功能、理解错业务规则
- 技术偏差：框架、数据库、ORM、鉴权、安全方案选错
- 结构偏差：分层错误、目录结构不合理、命名不一致
- 功能偏差：CRUD 不完整、接口缺失、前后端联调不通
- UI 偏差：风格不符、主题不符、布局不符、交互不符
- 文档偏差：README 缺失、部署说明不全、答辩报告不足
- 工程化偏差：Docker、构建脚本、CI、测试、环境变量缺失
- 质量偏差：硬编码密钥、异常处理缺失、校验不足、类型不匹配

## 总流程（6 个 Phase）

### Phase 1：问题接收与上下文定位

优先收集以下信息：
- 当前不符合需求的内容是什么
- 对应位置在哪里（模块/页面/API/文档/配置）
- 预期结果是什么
- 是否有参考需求文档、截图、报错、示例

如果用户描述过于模糊，先让用户在以下问题范围中选择：
- 功能不对
- 页面不对
- 接口不对
- 文档不对

### Phase 2：问题归类（必须先做）

收到问题后，先判断属于哪一类，可多选，但要给出“主归类”：

1) 需求理解类
- 与原需求不一致
- 业务规则遗漏
- 用户角色/权限理解错误
- 数据模型理解错误

2) 技术选型类
- 语言或框架不合适
- 数据库或 ORM 不合适
- 鉴权/权限方案不符合要求
- 工程化方案与环境不匹配

3) 后端实现类
- Controller 缺失或路由不对
- Service 逻辑不完整
- Mapper/Repository 查询错误
- Result/异常处理/校验不统一

4) 前端实现类
- 页面缺失或页面流程不完整
- 类型定义缺失或与接口不一致
- API 封装错误
- 路由、状态管理、表单交互有问题

5) UI 风格类
- 设计规范不符
- 配色方案不符
- 明暗主题不符
- 动画、字体、布局不符

6) 数据与安全类
- 表结构不合理
- DDL/迁移不完整
- JWT/权限/限流/参数校验缺失
- 存在 SQL 注入、硬编码密钥、敏感信息泄露风险

7) 工程化与交付类
- 启动失败或构建失败
- Docker/CI/CD 缺失
- README/部署文档/答辩报告不完整
- 测试覆盖不足

### Phase 3：定向追问（按归类选择题库）

每轮最多 4 个问题，只问与当前归类强相关的问题。若可一次确认，不要机械追问。

路由规则：
- 若当前只需要一级判断，使用本 Phase 的通用题库
- 若主类已足够明确，则停止在本 Phase 继续深问，切换到对应专项 Skill 处理

#### A. 需求理解类追问

Q1 偏差更接近哪种情况？
- 漏实现需求（推荐）
- 理解错需求
- 需求后来变更了
- 需求文档本身不清晰

Q2 影响范围？
- 单个模块（推荐）
- 多个模块
- 全局设计
- 还不确定

Q3 你希望的修复策略？
- 只补齐缺失项（推荐）
- 按新需求整体调整
- 先给修改方案再动代码
- 仅分析不修改

Q4 是否有参考依据？
- 需求文档（推荐）
- 截图/原型
- 示例系统
- 暂无参考

#### B. 技术选型类追问

Q1 需要调整的是哪一层？
- 后端技术栈
- 前端技术栈
- 数据库/ORM
- 构建/部署链路

Q2 修改强度？
- 最小兼容改造（推荐）
- 局部替换
- 模块级重构
- 整体重建

Q3 是否保留现有代码资产？
- 尽量保留（推荐）
- 可局部废弃
- 只保留接口协议
- 全部重来

Q4 优先目标？
- 快速可用（推荐）
- 更好维护性
- 更高性能
- 更低学习成本

#### C. 后端实现类追问

Q1 问题集中在哪一层？
- Controller
- Service
- Mapper/Repository
- 多层联动（推荐）

Q2 主要缺陷类型？
- CRUD 缺失（推荐）
- 业务规则错误
- 返回结构不统一
- 异常/校验缺失

Q3 数据访问方式需要怎样处理？
- 保持现状只修正逻辑（推荐）
- 补 ORM 映射
- 改 SQL 查询
- 重建数据层

Q4 安全修补范围？
- JWT
- 参数校验
- 权限控制
- 全部补齐（推荐）

#### D. 前端实现类追问

Q1 页面问题类型？
- 页面缺失
- 页面结构不对
- 交互流程不对（推荐）
- 接口联调不通

Q2 修改范围？
- 单页面
- 单模块多页面（推荐）
- 全局路由
- 全站样式

Q3 类型系统要求？
- 只补核心 interface
- 所有接口补全 TS interface（推荐）
- 增加运行时校验
- 暂不处理类型

Q4 优先修复哪类问题？
- 功能可用（推荐）
- 视觉一致
- 性能流畅
- 代码结构规范

#### E. UI 风格类追问

Q1 哪个维度最不符合？
- 设计规范
- 配色主题
- 布局模式
- 动画与字体

Q2 目标风格更接近？
- Material Design 3
- Ant Design
- Tailwind 风格（推荐）
- Bootstrap / 简洁企业风

Q3 主题模式要如何处理？
- 仅亮色
- 仅暗色
- 支持亮暗切换（推荐）
- 保持当前

Q4 改动策略？
- 只调样式变量（推荐）
- 调整组件外观
- 重做页面布局
- 连同交互动画一起重做

#### F. 数据与安全类追问

Q1 当前主要风险？
- 表结构设计问题
- 权限认证缺失
- 参数校验不足
- 敏感信息/注入风险（推荐）

Q2 处理方式？
- 最小修补（推荐）
- 安全基线补齐
- 全面安全加固
- 先列风险不动代码

Q3 需要覆盖哪些安全能力？
- JWT
- RBAC
- 限流
- 全部补齐（推荐）

Q4 数据变更方式？
- 直接改 DDL
- 增加迁移脚本（推荐）
- 只改模型定义
- 暂不改数据库

#### G. 工程化与交付类追问

Q1 当前痛点？
- 启动不了
- 打包不了
- 部署说明缺失
- 文档和测试不完整（推荐）

Q2 需要补哪些交付物？
- README
- Docker 配置
- CI/CD
- 全部补齐（推荐）

Q3 环境支持目标？
- 本地开发优先（推荐）
- Docker 优先
- 生产部署优先
- 教学演示优先

Q4 验证标准？
- 能启动即可
- 能构建 + 能运行（推荐）
- 通过测试
- 文档完整即可

### Phase 4：修改方案输出（改前确认）

在真正修改前，必须先输出“修改方案卡片”，至少包含：
- 主问题分类
- 次问题分类
- 影响范围
- 拟修改文件/模块
- 风险说明
- 修改策略（最小修复 / 局部重构 / 模块重做）
- 预期结果

然后询问用户选择：
- 直接修改（推荐）
- 先看详细方案
- 缩小修改范围
- 只分析不执行

### Phase 5：增量修改执行

执行规则：
- 优先局部改动，不破坏已正确部分
- 多文件修改时，按“接口定义 → 数据结构 → 业务逻辑 → 页面/文档”顺序推进
- 如果涉及需求补齐，先改核心链路，再补外围文档和样式
- 修改过程中要保留统一响应、统一异常、安全基线和命名规范

如果问题属于不同类别，建议拆成多个修改包：
- 修改包 A：后端逻辑
- 修改包 B：前端页面
- 修改包 C：UI 风格
- 修改包 D：工程化与文档

### Phase 6：回归检查与交付说明

修改完成后必须输出：

1) 修改对照表
- 问题项 | 归类 | 修改内容 | 修改位置 | 状态

2) 回归检查表
- 功能是否符合预期
- 前后端类型是否匹配
- 接口返回是否统一
- JWT/校验/权限是否正常
- UI 风格是否一致
- README / 脚本 / Docker / 文档是否同步更新

3) 未完成项追问

对仍未解决的项，必须提供选项：
- 继续补齐（推荐）
- 记录为下一轮修改
- 暂不处理
- 我来补充说明

## 问题归类判断规则（必须显式执行）

收到用户反馈后，先用下面规则做一次归类总结：

- 出现“少了、没做、遗漏、漏掉” -> 优先归到 `需求理解类` 或 `功能实现类`
- 出现“不是这个框架、数据库不对、技术栈不对” -> 归到 `技术选型类`
- 出现“接口报错、后端不返回、校验没生效” -> 归到 `后端实现类` 或 `数据与安全类`
- 出现“页面不好看、风格不一致、不是我想要的 UI” -> 归到 `UI 风格类`
- 出现“跑不起来、不能打包、不能部署” -> 归到 `工程化与交付类`
- 出现“文档不全、README 没写、答辩报告太差” -> 归到 `工程化与交付类`

如果一句反馈同时命中多个类别：
- 先选一个主归类
- 再列出 1–2 个次归类
- 只围绕主归类先问 2–4 题

## 与项目脚手架生成 Skill 的配合方式

如果当前项目由 `project-scaffold-generator` 生成，优先复用其原始信息：
- 原始需求分析表
- 原始技术选型结论
- 原始 UI 风格选择
- 原始 Quality Gate 结果

发现偏差时，先比较：
- 原需求 vs 当前实现
- 原确认项 vs 当前代码
- 原交付清单 vs 当前文件

若比较后能明确定位到具体层面，应立即路由：
- 页面、交互、路由、类型、联调 -> `frontend-fix-generator`
- Controller、Service、Repository、鉴权、异常 -> `backend-fix-generator`
- 表结构、DDL、迁移、索引、事务 -> `database-fix-generator`
- 配色、布局、主题、动效、字体 -> `ui-style-fix-generator`
- 构建、Docker、CI、测试、文档 -> `engineering-fix-generator`

## 质量底线

- 不随意覆盖用户已有正确修改
- 不引入硬编码密钥、密码、测试假数据污染
- 不为了“看起来完成”而写伪实现
- 不跳过回归检查
- 不在问题未确认时擅自大改技术栈

## 最终目标

让后期修改从“用户一句话 -> AI 盲改”升级为“问题归类明确 -> 追问聚焦 -> 修改可控 -> 回归可验收”的闭环流程。
