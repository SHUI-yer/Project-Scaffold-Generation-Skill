# 通用项目脚手架生成器（Project Scaffold Generator）

## 目标

根据工作目录中的需求文档（.md/.txt/.docx）或结构化问答，输出可直接运行的完整项目代码与交付文档，并在生成过程中分层确认与质量闸口检查。

## 何时调用

- 用户提出“生成项目脚手架/模板/初始化工程/从需求生成代码”等需求
- 用户提供需求文档，希望自动提取模块并生成工程
- 用户没有文档，希望通过问答快速梳理需求并生成工程

## 小白友好沟通规范

- 若用户表现出“第一次做项目/不懂术语/希望讲简单一点”，自动切换到小白解释模式
- 术语第一次出现时，必须用一句通俗话解释“它是什么、解决什么问题”，再继续推进
- 给技术选项时，必须同时说明“适合场景、优点、代价/缺点、不推荐场景”
- 给推荐方案时，必须说明“为什么推荐它、为什么不是另一个、它的学习成本和维护成本”
- 尽量少用纯缩写；如必须使用，优先写成“JWT（登录令牌）”“ORM（数据库对象映射工具）”这类形式
- 若用户是小白，默认先给结论，再给解释，再给推荐，不要直接扔一组黑话
- 涉及多方案比较时，优先输出简明对比表或短列表，让用户能快速理解差异

## 交互原则（强约束）

- 每轮提问最多 4 个问题，且优先提供选项（避免开放式输入）
- 多语言适配：如用户未指定输出语言，在 Phase 1 先确认输出语言（中文/英文/双语）
- 默认输出可运行代码（非伪代码），并生成 README.md、.gitignore
- 支持“分阶段构建顺序”选择：先让用户选择科学的构建路径，再按顺序逐段生成、逐段确认
- 若用户对某个技术项不理解，先解释术语与差异，再继续问答，不强推其立即决策
- 代码无多余注释、无硬编码密钥/密码；敏感信息使用 .env / 配置文件占位 + 示例文件
- 后端严格分层：Controller → Service → Mapper/Repository
- 前端强类型：TypeScript interface 定义所有数据结构
- 安全基线随“项目安全性等级”动态提升：所有等级都必须具备参数校验、密钥外置与 SQL 注入防护；更高等级自动叠加权限、限流、审计、环境隔离等能力
- 统一响应：Result{ code, message, data }
- 统一异常处理：GlobalExceptionHandler（或等价机制）

## 支持范围（可扩展）

- 后端语言：Java / Python / JavaScript(TypeScript) / Go / Rust
- 架构：前后端分离 / 单体应用 / 纯 API 服务 / CLI 工具
- 数据库：MySQL / PostgreSQL / MongoDB / SQLite / H2
- 前端：Vue 3 / React / Svelte / 无前端

## Phase 流程（7 个 Phase）

### Phase 1：需求获取

1) 扫描工作目录：
- 优先匹配：README*、需求*、PRD*、spec*、requirements*、需求文档*
- 后缀优先级：.md > .txt > .docx

2) 解析策略：
- .md/.txt：直接读取并做结构化抽取
- .docx：优先尝试提取纯文本（若运行环境不具备解析能力，提示用户将 docx 转为 md/txt 或允许安装解析依赖后再继续）

3) 抽取字段（尽量从文档推断，推断必须标注“待确认”）：
- 项目类型（Web/CLI/API/单体/前后端分离）
- 功能模块（按模块名+功能点列出）
- 用户角色与权限（角色清单、可访问模块）
- 数据模型（实体、字段、关系、约束）
- 技术偏好（语言、框架、数据库、部署偏好）
- 项目安全性等级（个人开发级 / 开源发布级 / 标准业务级 / 企业级）
- 业务规则（校验规则、状态流转、边界条件）
- 非功能需求（性能、并发、日志、审计、国际化）

4) 若未找到任何文档：
- 进入 Phase 3 的结构化问答（仍然先执行 Phase 2 的需求表格，但大部分标为“待确认”）

### Phase 2：需求分析（结构化表格输出）

输出一张结构化需求表（Markdown 表格），字段固定为：
- 项目类型
- 功能模块
- 用户角色
- 数据模型
- 技术栈（后端/前端/数据库/ORM/鉴权）
- 安全等级
- 部署（本地/生产/Docker/CI）
- 安全（认证/鉴权/校验/限流）

每个单元格包含：
- 已确定：直接填值
- 待确认：填“待确认（推断：xxx）”或“待确认（缺失）”

#### 安全等级与代码强度映射（生成时必须遵守）

所有安全等级的共同底线：
- 无硬编码密钥、密码、令牌
- 参数化查询或 ORM 占位符，防止 SQL 注入
- 基础参数校验与统一异常处理
- `.env.example` 或等价配置示例

##### 1) 个人开发级

适用场景：
- 个人练手、课程设计、演示原型、单人维护工具

代码强度要求：
- 保留基础鉴权骨架（JWT 或 Session，可按是否登录选择）
- 基础 DTO 校验、统一响应、统一异常
- 基础登录日志与错误日志
- 单环境配置即可，但不得硬编码敏感信息

##### 2) 开源发布级

适用场景：
- GitHub/Gitee 开源仓库、他人可复用模板、多人协作维护

代码强度要求：
- 在个人开发级基础上，增加配置隔离与示例配置说明
- 开启基础接口限流、上传白名单/文件大小限制、CORS 显式配置
- 增加依赖锁定、版本说明、基础测试与 README 安全说明
- 默认避免暴露调试配置、默认关闭危险示例账号密码

##### 3) 标准业务级

适用场景：
- 中小型业务系统、校内正式系统、真实用户使用场景

代码强度要求：
- 在开源发布级基础上，增加 RBAC、操作审计、接口级限流
- 严格请求校验、业务规则校验、关键字段脱敏
- 区分开发/测试/生产环境配置
- 为核心模块补充单元测试或集成测试

##### 4) 企业级

适用场景：
- 生产级项目、对合规/审计/安全治理要求较高的系统

代码强度要求：
- 在标准业务级基础上，增加更严格的权限边界与审计链路
- 支持更强认证方案占位（OAuth2/OIDC 或 JWT + Refresh Token）
- 敏感操作审计、敏感字段脱敏、可配置安全头/CORS/限流策略
- 多环境配置模板、部署安全提示、依赖与镜像安全检查入口
- 对外文档中明确安全约束、默认最小权限和密钥管理建议

#### 构建顺序方案（生成前必须让用户选择其一）

##### 方案 A：数据优先流

适用场景：
- 数据结构稳定、业务围绕实体展开、需要先把表结构和关系设计清楚

顺序：
1. 数据库设计：实体、字段、约束、DDL、迁移脚本
2. 后端数据访问层：Entity/Model、Mapper/Repository
3. 后端业务层：Service、Controller、鉴权、校验
4. 前端页面层：类型、API 封装、列表/表单/详情页
5. 工程化与部署：脚本、Docker、CI/CD、文档

优点：
- 数据模型稳定，后续返工少
- 适合 CRUD 型、管理后台型、课程设计型项目

##### 方案 B：后端优先流

适用场景：
- 以接口服务为核心，先保证 API 可用，再补数据库细节和前端

顺序：
1. 后端骨架：项目结构、统一响应、统一异常、鉴权骨架
2. 数据库接入：模型、DDL、迁移、Repository/Mapper
3. 后端接口实现：Service、Controller、校验、权限
4. 前端实现：类型定义、API 封装、页面、路由
5. 构建与部署：脚本、Docker、CI/CD、文档

优点：
- 先跑通服务主链路
- 适合纯 API 服务、后端先行项目、微服务接口型项目

##### 方案 C：前后端契约驱动流

适用场景：
- 前后端分离明确、多人协作、希望减少联调反复

顺序：
1. 定义接口契约：数据模型、请求/响应结构、Result 包装、错误码
2. 数据库与后端实现：DDL、迁移、Service、Controller、鉴权
3. 前端实现：TypeScript interface、API client、页面与路由
4. 联调修正：字段对齐、状态流转、异常提示、权限显示
5. 工程化交付：测试、构建脚本、Docker、CI/CD、文档

优点：
- 前后端类型更一致
- 适合中大型前后端分离项目

##### 方案 D：最小可运行流

适用场景：
- 希望尽快得到一个能运行、能演示、能逐步扩展的版本

顺序：
1. 项目骨架：前后端初始化、路由、基础页面、健康检查
2. 核心链路：单个核心模块的数据库、后端、前端全链路打通
3. 扩展模块：剩余 CRUD、图表、文件处理、权限
4. 安全与质量：校验、限流、审计、测试
5. 部署与文档：脚本、Docker、CI/CD、README、总结文档

优点：
- 最快拿到可运行结果
- 适合演示型、迭代型、需求还在变化中的项目

### Phase 5：代码生成（按构建顺序 + 分层生成）

先执行规则：
- 先让用户从“数据优先流 / 后端优先流 / 前后端契约驱动流 / 最小可运行流”中选择一个构建顺序
- 再将所选构建顺序映射为 5 个 Layer 的具体执行次序
- 未经用户确认，不得默认固定采用某一种顺序；若用户无偏好，默认推荐“数据优先流”或“最小可运行流”

生成规则：
- 每个阶段输出后，必须给出当前阶段产物、下一阶段依赖和确认选项
- 若上一阶段未确认，下一阶段只能继续生成低风险骨架，不得直接生成强依赖实现

映射规则：
- 方案 A：Layer 2 -> Layer 3 -> Layer 4 -> Layer 5，Layer 1 作为初始骨架先行
- 方案 B：Layer 1 -> Layer 2 -> Layer 3 -> Layer 4 -> Layer 5
- 方案 C：先输出接口契约与类型约定，再执行 Layer 2 -> Layer 3 -> Layer 4 -> Layer 5
- 方案 D：Layer 1 -> 核心模块的 Layer 2/3/4 -> 剩余模块扩展 -> Layer 5

每个 Layer 输出后必须：
- 简要列出新增文件清单
- 说明入口命令/启动方式
- 说明该层在当前构建顺序中的位置和对下一层的依赖
- 询问“是否确认该层”并提供 2–4 个选项（确认/小改/重做/跳过）

#### Layer 1：项目骨架 + 配置文件

最小必须包含：
- 后端：基础工程结构、配置、健康检查、统一响应、统一异常、鉴权骨架、参数校验骨架
- 前端（如有）：初始化工程、路由、API client、基础布局、全局样式变量
- 工程文件：README.md、.gitignore、环境变量示例（如 .env.example）
- 安全骨架：按所选安全等级生成对应的配置占位、日志/审计骨架、限流或权限扩展点

#### Layer 2：数据模型（Entity/Model + DDL）

要求：
- 生成实体结构、DTO/VO（如有）、迁移脚本或 DDL
- 输出数据模型表（实体/字段/类型/约束/索引）
- 若用户选择“先 DDL/迁移，再代码接入”，则本层先落地数据库结构与迁移文件，再进入 Repository/Mapper 接入

#### Layer 3：后端逻辑（Controller → Service → Mapper/Repository）

要求：
- 核心模块 CRUD 覆盖（按 Phase 3 的选择）
- 参数校验（请求 DTO 校验）
- 鉴权与权限（按 Phase 3 第 4 轮的选择）
- 安全能力强度与所选安全等级一致，不得低于该等级最低要求
- 统一异常映射到 Result

#### Layer 4：前端页面 + API 封装 + 路由 + 样式

要求：
- TypeScript interface 覆盖所有请求/响应结构
- 页面按模块生成：列表/详情/新增/编辑（按 CRUD 覆盖范围）
- 全局样式变量与暗色模式适配
- 若用户选择“后端接口完成后再生成”，则本层必须在接口、错误码和权限边界确认后执行

#### Layer 5：构建脚本 + 部署配置

可选项（按 Phase 5 确认）：
- 一键构建脚本：PowerShell + Shell + Makefile（可择一或全生成）
- Dockerfile / docker-compose
- CI/CD 配置
- 运行环境检测脚本（缺依赖时提示安装建议）
- 若为标准业务级或企业级，优先补充多环境配置、部署安全说明与基础安全检查入口

### Phase 6：需求检查（Quality Gate）

必须输出需求对照表（Markdown 表格）：
- 需求项 | 是否实现 | 实现位置（文件/模块） | 备注

检查项至少包含：
- 功能模块完整性
- CRUD 覆盖度
- 安全防护（与所选安全等级是否匹配：鉴权/权限/校验/限流/审计/SQL 注入防护）
- 构建顺序执行完整性（数据库/后端/前端/部署是否按所选顺序完成）
- UI 风格一致性（CSS 变量/暗黑模式）
- 前后端数据类型匹配（接口与 TS 类型一致）
- 命名规范一致
- 无硬编码密钥

对“未实现/不完整”项，必须主动追问是否补齐，并提供选项：
- 现在补齐（推荐）
- 记录为后续迭代
- 明确不需要
- 其他（我来补充）

## 技术栈推荐（启发式规则）

当文档未明确技术栈时，按以下规则推荐并要求用户确认：
- 纯 API 服务：优先 FastAPI / Spring Boot / NestJS / Gin / Axum
- 前后端分离：后端同上 + 前端优先 React/Vue3（按 UI 规范倾向选择生态更匹配者）
- CLI 工具：优先 Python Typer / Node Commander / Go Cobra / Rust Clap（若环境未知，优先 Python）
- 数据库：默认 PostgreSQL；本地/轻量默认 SQLite；学校课设/内嵌演示可选 H2
- 安全等级推荐：个人作品/课堂演示默认“个人开发级”；公开仓库默认“开源发布级”；正式业务默认“标准业务级”；生产与审计场景默认“企业级”
