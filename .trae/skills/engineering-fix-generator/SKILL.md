# 工程化专项修改生成器（Engineering Fix Generator）

## 目标

当问题集中在项目运行、构建、部署和交付层时，围绕环境变量、构建脚本、Docker、CI/CD、测试、文档和依赖治理进行专项分析与增量修复。

## 何时调用

- 用户反馈“跑不起来”“打包失败”“Docker 不对”“CI 没配”“README 不全”
- `project-fix-generator` 将主归类识别为 `工程化与交付类`
- 功能代码基本存在，但工程链路和交付物不完整

## 小白友好沟通规范

- 若用户不理解“Docker、CI/CD、构建脚本、环境变量、依赖治理”这些词，首次出现时必须解释
- 解释工程问题时，优先用“怎么启动、怎么打包、怎么部署、哪里卡住了”这类过程语言
- 比较方案时，要说明“学习成本、上手速度、自动化程度、后续维护难度”
- 给推荐策略时，必须说明为什么先修环境、先补脚本还是先补 Docker/CI
- 尽量避免只给命令名，不解释它在整个交付流程中的作用

## 核心原则

- 先识别是运行环境、依赖、构建、部署、测试还是文档问题
- 构建链路优先于扩展文档
- 优先提供可复现、可执行、可验证的工程产物
- 不把临时手工步骤当作正式交付方案
- 若用户是小白，默认补充“术语解释 + 流程说明 + 推荐修复路径”

## 专项问题分类

1) 环境与依赖类
- 版本不兼容
- 依赖缺失
- 包管理器不一致
- 本地与生产环境差异大

2) 构建脚本类
- 启动脚本缺失
- 打包脚本错误
- 多模块构建顺序不合理
- 环境检测脚本缺失

3) Docker 与部署类
- Dockerfile 不完整
- docker-compose 缺依赖服务
- 端口/卷/网络配置错误
- 生产部署步骤不清晰

4) CI/CD 与质量类
- CI 不触发
- 测试链路缺失
- lint/build/test 顺序不合理
- 发布流程不清楚

5) 文档与交付类
- README 缺失
- 环境说明不足
- 配置示例不足
- 项目总结/答辩报告缺失

## 修改顺序

1. 明确目标运行环境
2. 对齐依赖与版本矩阵
3. 修复构建/启动脚本
4. 修复 Docker/CI/CD 配置
5. 补齐 README、部署说明和交付文档

## 回归检查

- 本地开发环境可复现
- 构建脚本可执行
- Docker 配置能启动核心依赖
- CI 流程逻辑完整
- README 与配置说明可指导新用户运行
- 交付文档与当前代码一致
