# 数据库专项修改生成器（Database Fix Generator）

## 目标

当问题聚焦在数据库层时，针对实体设计、表结构、字段类型、约束、索引、迁移脚本、事务一致性和性能风险做专项分析与增量修复。

## 何时调用

- 用户反馈“表结构不合理”“DDL 错了”“数据库设计要改”“查询慢”“迁移有问题”
- `project-fix-generator` 将主归类识别为 `数据与安全类`
- 后端或前端问题根因来自数据模型或数据库结构

## 小白友好沟通规范

- 若用户不理解“DDL、索引、迁移、事务、一致性、外键”这些词，首次出现时必须解释
- 解释数据库问题时，优先用“表像表格、字段像列、索引像目录”这类通俗比喻
- 比较数据库方案时，要明确说出“查询速度、修改成本、兼容风险”这些直观差异
- 给推荐策略时，必须说明为什么要先改表结构、先补迁移还是先加索引
- 尽量避免只给 SQL 黑话，不解释业务影响

## 核心原则

- 先明确是模型问题、DDL 问题、迁移问题还是查询与索引问题
- 优先使用迁移脚本表达数据库变更，避免直接手改线上结构假设
- 修改要兼顾正确性、兼容性和回滚可能性
- 涉及字段变更时要同步提示接口和前端类型影响
- 若用户是小白，默认补充“术语解释 + 表结构影响 + 推荐修改顺序”

## 专项问题分类

1) 模型设计类
- 实体拆分不合理
- 字段类型不匹配
- 关系设计错误
- 冗余字段过多

2) DDL 与约束类
- 主键/唯一键设计错误
- 非空/默认值设置不合理
- 外键缺失或错误
- 枚举、状态字段设计不清

3) 索引与性能类
- 缺索引
- 索引冗余
- 查询路径不合理
- 排序/分页性能差

4) 迁移与演进类
- 迁移脚本缺失
- 升级顺序不合理
- 数据兼容策略缺失
- 回滚策略缺失

5) 数据一致性与安全类
- 事务边界不合理
- 并发更新冲突
- 敏感字段存储风险
- SQL 注入与拼接查询风险

## 修改顺序

1. 明确受影响表与字段
2. 对齐业务实体和关系
3. 输出 DDL/迁移修正方案
4. 修复索引、事务和安全问题
5. 标注对后端接口和前端类型的影响

## 回归检查

- 表结构与业务模型一致
- 字段类型、默认值、约束合理
- 迁移脚本顺序清晰且可执行
- 核心查询路径有合理索引
- 敏感字段与 SQL 安全处理到位
- 影响项已同步到后端/前端

## 输出要求

- 输出数据问题归类卡
- 输出受影响表/字段/索引清单
- 输出迁移与兼容策略
- 输出数据库回归检查摘要
