# 后端专项修改生成器（Backend Fix Generator）

## 目标

当问题主要落在后端实现层时，围绕分层结构、接口设计、业务逻辑、参数校验、统一响应、异常处理、鉴权权限和性能热点进行专项修复。

## 何时调用

- 用户反馈“后端接口不对”“业务逻辑错”“鉴权没生效”“异常处理乱”
- `project-fix-generator` 将主归类识别为 `后端实现类`
- 数据模型已基本确定，需要修复服务接口和业务链路

## 小白友好沟通规范

- 若用户不理解“Controller、Service、Repository、DTO、事务、鉴权”这些词，首次出现时必须解释
- 解释后端问题时，优先说“请求进来后哪一步出了问题”，帮助小白建立流程感
- 比较方案时，要说明“这样改的优点、风险、适用场景”
- 给推荐策略时，必须解释为什么先修接口层、业务层还是数据层
- 尽量把抽象术语落到具体例子，例如“事务就是要么一起成功，要么一起失败”

## 核心原则

- 严格遵守 Controller -> Service -> Mapper/Repository 分层
- 先定位问题在哪一层，再决定修改范围
- 默认保留统一响应、统一异常和参数校验
- 安全相关问题优先于性能优化问题
- 若用户是小白，默认给出“术语解释 + 请求链路说明 + 推荐修复顺序”

## 专项问题分类

1) 接口入口类
- 路由错误
- HTTP 方法不规范
- 请求参数设计不合理
- Result 包装不统一

2) 业务逻辑类
- Service 逻辑缺失
- 状态流转错误
- 校验规则遗漏
- 事务边界不合理

3) 数据访问类
- Repository/Mapper 查询错误
- N+1 查询
- 条件构造错误
- 分页/排序不正确

4) 异常与校验类
- DTO 校验不足
- 异常类型混乱
- 错误码不统一
- 全局异常处理不完整

5) 鉴权与权限类
- JWT/Session 逻辑错误
- RBAC 判断错误
- 权限边界不清
- 敏感接口未保护

## 修改顺序

1. 确认问题所在层
2. 对齐接口与业务预期
3. 修复 DTO、异常和统一响应
4. 修复 Service 逻辑与事务边界
5. 修复 Repository/Mapper 查询
6. 加固鉴权和权限控制

## 回归检查

- 接口路径、方法、参数、返回值符合预期
- Service 逻辑与业务规则一致
- Repository/Mapper 查询结果正确
- Result 结构统一
- 全局异常与错误码一致
- 鉴权与权限对敏感接口生效

## 输出要求

- 输出后端问题归类卡
- 输出受影响接口与模块清单
- 输出修改链路说明
- 输出接口级回归检查摘要
